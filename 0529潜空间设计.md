# 0529 Video-OPD 潜空间推理框架设计（与 Coconut 的差异 + SFT/OPD 两阶段方案）

> 本文档目的：
> 1. 明确我们的潜空间方案与原版 Coconut **不同**的地方（不是照抄）；
> 2. 划清 **SFT 阶段** 与 **OPD 阶段** 在潜空间训练上的根本差别；
> 3. 解决推理时一个核心难题：**think 内部下一个位置应该再产出潜空间特征，还是该输出 `</think>` 结束思考？**

---

## 〇、术语约定

- **潜空间特征 (latent feature, z)**：学生 transformer 最后一层在某个位置的 hidden state，**不被解码成具体 token**，而是被当作"连续思维"。
- **解码器 (Decoder LM head)**：学生模型初始 LM head 的**冻结副本**。负责把 latent z 投影回 vocab logits，用来做训练监督 / 推理时的"读出"。
- **trajectory**：训练样本里的 GT 推理链，结构固定为：

  ```text
  <think>
  [Analyze] ...
  <observe type="..." .../>
  <result>...</result>
  [Conclude] ...
  </think>
  <answer>...</answer>
  ```

- **学生最终输出格式（恒定）**：

  ```text
  <think>
  <latent × N>      ← think 内部全部是 latent，没有任何显式控制符 token
  </think>
  <answer>...</answer>
  ```

  其中 N 为 **奇数**：N = 2k+1（k ≥ 0）。`<observe/>`、`<result>...</result>` 等 GT trajectory 里的控制符号**不出现在学生 token 流中**，只存在于「冻结 Decoder 把 latent 解码出来给外部系统看」的过程里。
- **奇数 latent 块（第 1、3、5… 个）= 推理段**：对应 GT 里的 `[Analyze] ...` / `[Conclude] ...`，由 **Teacher_R**（推理教师）监督。
- **偶数 latent 块（第 2、4… 个）= 感知段**：对应 GT 里的 `<result>...</result>`，由 **Teacher_P**（感知教师）监督。
- **块切换规则**：每个奇数块结束后，由冻结 Decoder 解码出该块的末尾是否以 `/>` 结尾 → 是则进入下一个偶数块（感知），不是则本块即为最后一块（`[Conclude]`收尾）。每个偶数块结束后无条件进入下一个奇数块。

---

## 一、与 Coconut 的关系与差异

### 1.1 Coconut（FAIR 2024.12）原版

- think 区域内**完全没有 token 监督**，hidden state 自由演化作为 next inputs_embeds；
- 仅对最终 answer 区域做 CE；
- 通过分阶段 schedule 把显式 CoT 步骤逐步替换为 continuous thought。

### 1.2 我们的设计（Video-OPD）—— "结构化潜空间"

我们**不是**纯 Coconut，因为：

1. 我们的 think 内部有**结构化控制符号**（`<observe/>`、`<result>...</result>`、`</think>`），它们必须显式出现，不能潜空间化 —— 否则训练完模型不知道何时调用工具、何时结束思考。
2. 我们在 SFT 阶段**有完整 GT 推理链**，"潜空间区段"在 GT 里是确定的（`[Analyze]` / `[Conclude]` 段），可以混合使用「潜空间训练 + 文本 CE」。
3. OPD 阶段**没有 GT 推理链**（只有最终答案），此时才退化为接近 Coconut 的"自由潜空间"。

### 1.3 一句话总结差异

> Coconut 全 think 都是 latent；我们 think 内部 **`[Analyze]` / `[Conclude]` 段是 latent，`<observe.../>` / `<result>...</result>` / `</think>` 是显式 token**。

学生 token 流只有 4 类位置：`<think>` 显式 / latent / `</think>` 显式 / `<answer>...</answer>` 显式。其中 latent 区段进一步按

## 三、SFT 阶段 —— 我们**知道**每个 latent 块的位置、长度和 GT 内容

### 3.1 关键事实

SFT 阶段 GT trajectory 完整已知，我们可以从 GT 切出每个 latent 块的范围：

- **奇数块的 GT 文本** = GT trajectory 中 `[Analyze]` 或 `[Conclude]` 段（含末尾可能的 `<observe .../>` 收尾）；
- **偶数块的 GT 文本** = GT trajectory 中 `<result>...</result>` 内部内容；
- **块数（奇数）和每块长度**都由 GT 直接给出。

以样本 B（1 次 observe）为例，GT 切片如下：

| 块编号 | 奇/偶 | GT 文本（latent 内部解码目标） |
|---|---|---|
| 1 | 奇（推理）| `[Analyze] Need to localize when "..." happens.\n<observe type="temporal_locate" target="..."/>` |
| 2 | 偶（感知）| `It happens from 0.0s to 13.6s.` |
| 3 | 奇（推理）| `[Conclude] Time span confirmed.` |

样本 A（0 次 observe）只有 1 个奇数块：

| 块编号 | 奇/偶 | GT 文本 |
|---|---|---|
| 1 | 奇（推理）| `[Analyze] What did the person do with the food?\n[Reason] Based on the video content, I can determine the answer.` |

### 3.2 SFT 训练 forward 设计（修正版）

对于每个 batch，构造 4 个 mask：

- `latent_odd_mask[B, L]`：标记奇数块位置（推理段，由 Teacher_R / GT 推理文字监督）；
- `latent_even_mask[B, L]`：标记偶数块位置（感知段，由 Teacher_P / GT result 监督）；
- `text_mask[B, L]` = 其余需算 loss 的显式 token 位置（`<think>`、`</think>`、`<answer>...</answer>`）；
- `loss_mask[B, L]` = 三者并集（即所有 labels != -100 的位置）。

**单次 forward + 双 head 路由**：

```python
outputs = student(input_ids, ..., output_hidden_states=True)
last_hidden = outputs.hidden_states[-1]              # (B, L, D)

logits_student = outputs.logits                       # 学生自己的 head（用于显式 token 段）
logits_decoder = frozen_decoder_lm_head(last_hidden)  # 冻结 head 解码 latent（奇+偶都用这一个）

latent_mask = latent_odd_mask | latent_even_mask
combined_logits = torch.where(
    latent_mask.unsqueeze(-1),
    logits_decoder,    # latent 段 → 冻结 Decoder
    logits_student,    # 显式 token 段 → 学生 head
)
loss = CE(combined_logits[:, :-1], labels[:, 1:], ignore_index=-100)
```

> **奇/偶块为什么用同一个 Decoder？** 因为冻结 Decoder = 学生初始 LM head 的拷贝，本身就是一个通用语言投影。奇/偶的

## 四、OPD 阶段 —— 我们**不知道**块数也不知道每块长度，全由学生自主决定

### 4.1 关键事实

OPD 阶段：

- 只有原始问题 + 最终答案（或 verifiable reward）；
- 没有 GT trajectory，因此**不知道学生应该产生几个 latent，也不知道何时该 `<observe/>`，何时该 `</think>`**；
- 学生必须**自主决定**："我下一步还要继续 think（输出 latent），还是该 `<observe/>` 调外部工具，还是该 `</think>` 结束并 `<answer>`？"

### 4.2 核心问题

> **每个 latent 块何时结束？结束之后是该进入下一个 latent 块（继续 think），还是该吐 `</think>` 收尾？**

### 4.3 解法（基于「奇数块末尾是否以 `/>` 结尾」判定）

**核心规则**：是否进入下一个 latent 块，由当前块**冻结 Decoder 解码出来的文本末尾**判定，不需要背控制符表。

定义 `EOB`（end-of-block）信号：在某个 latent step 上，冻结 Decoder 预测出的 token 是 `eos_token_id` （或任意选定的块结束符，比如训练时使用的 `\n` 表示一句推理完成）。

**生成状态机**：

```
block_idx = 1                 # 从第 1 个奇数块（推理）开始
prev_embed = embed("<think>")
emit_token("<think>")

while block_idx <= MAX_BLOCKS:
    # ===== 走完一个 latent 块：一直 step 到出现 EOB =====
    block_logits_seq = []
    block_hidden_seq = []
    for step in range(MAX_LATENT_STEPS_PER_BLOCK):
        z_t = student.forward_one_step(prev_embed)        # hidden state，不解码为 token
        decoded_logits = frozen_decoder(z_t)
        decoded_token = argmax(decoded_logits)
        block_logits_seq.append(decoded_logits)
        block_hidden_seq.append(z_t)
        prev_embed = z_t                                   # latent 回注（Coconut 式）
        if decoded_token == EOB:
            break

    # ===== 解码本块，看奇数块末尾是否以 "/>" 结尾 =====
    block_text = decode(block_logits_seq)                  # 仅用于状态判断/外部工具调用
    if block_idx 是奇数:
        if block_text.rstrip().endswith("/>"):
            # 该块结尾是一个 observe 调用 → 调用工具，清理后进入下一个（偶数）块
            observe_call = parse_observe(block_text)
            tool_artifact = run_tool(observe_call)
            # 注：工具产物不会作为显式 token 进入学生输入，
            # 而是靠下一个偶数块在 latent 中「代表」该感知结果（由 Teacher_P KD 出来）
            ctx_for_next_block = tool_artifact              # 进入下一个偶数块时，可选地把 tool_artifact 接入视觉输入
            block_idx += 1
        else:
            # 该块是收尾 [Conclude] 推理，合理结束
            break
    else:  # block_idx 为偶数 = 感知块
        # 感知块完成后必然进入下一个奇数块（继续推理或收尾）
        block_idx += 1

emit_token("</think>")
# 进入 <answer>...</answer> ，此后完全走学生自己 head 的显式 token 自回归
```

**用人话讲**：

- 每个 latent 块走到 EOB（冻结 Decoder 吐出句末符 / EOS）就停；
- 取该块解码出来的文本，看末尾：
  - **奇数块以 `/>` 结尾** → 该块以一个 `<observe .../>` 收尾 → 需要进入下一个偶数块（感知），所以下一个偶数块必然出现；
  - **奇数块不以 `/>` 结尾** → 该块只有 `[Conclude]`、没有新 observe 调用 → 推理结束，吐 `</think>` 进入 answer；
  - **偶数块**（感知块）**不需要看末尾**，走完必然进入下一个奇数块（由学生决定那个奇数块是继续 observe 还是收尾）。
- **未出现过「奇数块中间吐 `/>`」的场景**：SFT 阶段 GT 训练保证了 `<observe .../>` 只出现在奇数块末尾、不可能出现在偶数块。

### 4.4 OPD 训练 forward 设计

OPD 阶段没有 GT trajectory，所有监督都通过教师 logits 投放到「冻结 Decoder 解码后的位置」上：

1. 学生按 §4.3 状态机自主 rollout，产生交替的奇数/偶数 latent 块。
2. 对**奇数块**（推理段）：调用 Teacher_R 处理「问题 + 已有 R_1…R_{i-1}」，让 Teacher_R 在文本空间逐步生成对应 R_i 的 logits 序列；学生该块每个 latent step 经冻结 Decoder 输出的 logits，与 Teacher_R 对应位置的 logits 做 **KL**。
3. 对**偶数块**（感知段）：用上一奇数块末尾解析出的 `<observe .../>` 调真实工具，得到工具产物 + 子任务；喂给 Teacher_P，Teacher_P 输出该感知段 R_i 的 logits 序列；学生偶数块的冻结 Decoder logits 与 Teacher_P logits 做 **KL**。
4. **块边界（EOB）的判定标记**本身也由教师供信：Teacher_R / Teacher_P 在对应文本末尾位置给出的 EOS 高概率就是学生该 step 的期望输出。
5. 最终 `<answer>` 显式 token 段：标准 CE 到 GT 答案，或 verifiable reward 反传。

> 学生在 latent step 上的 hidden state **既要能被冻结 Decoder 解码出接近教师 logits**（KL 反传重塑 hidden），**又要作为下一 step 的 inputs_embeds 能被学生本体理解**。这是 OPD 阶段唯一的 hidden 供给点。

### 4.5 状态机的健壮性保障

| 风险 | 应对 |
|---|---|
| 单个 latent 块无限不结束（EOB 不出现）| 硬上限 `MAX_LATENT_STEPS_PER_BLOCK = 64`，超过则强制 EOB |
| 整条 trajectory 块数过多 | 硬上限 `MAX_BLOCKS = 9`（即最多 4 次 observe），超过则强制吐 `</think>` |
| 奇数块解码出来有 `<observe` 但缺 `/>` | 启发式补全重试一次；仍不合法则判为错误，本轮偶数块 KL 跳过，推理提前收尾 |
| 工具调用失败 / observe 解析失败 | 复用 `parse_student_observe_strict` + `ObserveParseError`，该偶数块 KL 跳过，但后续奇数块 KL 仍可以走 |
| 冻结 Decoder 解码不准（训练初期）| SFT 阶段已经把 latent 锁在语言空间，OPD 接续初期不会太离谱 |

## 五、SFT 与 OPD 的关键差异对照表

| 维度 | SFT 阶段 | OPD 阶段 |
|---|---|---|
| 是否有 GT trajectory | ✅ 完整 | ❌ 仅有最终答案 / reward |
| latent 块个数 | 由 GT 推出（§3.1）：0 次 observe = 1 块；1 次 observe = 3 块 | 由状态机动态决定（1 / 3 / 5 / 7 / …）|
| 单块 latent 长度 | = 该块 GT 文本 tokenize 后的长度 | 由冻结 Decoder 在某 step 预测出 EOB 动态决定 |
| latent hidden 是否回注为 next inputs_embeds | 训练时不需要（输入是 GT token）；推理时需要 | 训练 + 推理时**都需要** |
| 奇数块 loss 来源 | 冻结 Decoder 解码后 CE 到 GT 推理文本 | 冻结 Decoder logits 与 Teacher_R logits 做 KL |
| 偶数块 loss 来源 | 冻结 Decoder 解码后 CE 到 GT result 文本 | 冻结 Decoder logits 与 Teacher_P logits 做 KL |
| 何时产生 `</think>` | GT 中 `</think>` 位置直接告知 | "末尾奇数块不以 `/>` 结尾"时动态产生 |
## 六、实现路线图（基于现状的增量改造）

### 6.1 当前代码已有的（不动）

- `LatentSpaceForward`（`training/stage1_sft_train.py:395`）：双 head 路由框架；
- `compute_think_mask`：定位 `<think>...</think>` 区间；
- `create_decoder_lm_head`：冻结副本；
- OPD 端 `parse_student_observe_strict` + 感知段 KL。

### 6.2 需要新增 / 改造的

1. **`compute_latent_odd_even_mask`（取代当前粗粒度 `think_mask`）**
   - 当前 mask 把整个 `<think>...</think>` 都标 True（包括显式控制符），不准；
   - 新版 mask 要**同时生成奇偶两张**：
     - `latent_odd_mask`：包含所有奇数块内部位置（`[Analyze]...<observe.../>` 整体，含末尾 `/>`）；
     - `latent_even_mask`：包含所有偶数块内部位置（`<result>` 与 `</result>` 之间的纯内容）；
     - 两者互斥，且并集不包括 `<think>` / `</think>` / `<answer>...</answer>` / 间隔换行。

2. **推理状态机（新文件 `inference/latent_generate.py`）**
   - 实现 §4.3 状态机；EOB 判定、奇数块末尾 `/>` 检查、偶数块直返下一轮；

3. **OPD 训练 rollout 逻辑（`training/stage2_opd_train.py`）**
   - 复用 `latent_generate` 状态机得到学生的奇/偶块 hidden 序列；
   - 调用 Teacher_R 生成奇数块文本 logits、调用工具 + Teacher_P 生成偶数块文本 logits；
   - 逐位 KL 反传到学生。

4. **EOB 信号选择**
   - 默认 `EOB = eos_token_id` 或者训练使用的 `\n` token；
   - 在 `configs/latent_eob.py` 中集中配置，训练与推理保持一致。

### 6.3 测试顺序

1. ✅ `infer_from_ckpt.py` 支持随机抽样（已完成）；
2. ⏳ 拆分 `latent_odd_mask` / `latent_even_mask`，重训 SFT；
3. ⏳ 在新 ckpt 上加 `--latent_inference`，验证状态机能否正确在奇/偶块间切换；
4. ⏳ 在该 ckpt 上跑 OPD（状态机 rollout + 双教师 KL）；

## 九、数据样例完整拆解（学生 / Teacher_R / Teacher_P 各学什么）

> 本节用**当前 SFT 数据集中真实存在的两类样本**做端到端拆解。
>
> 范围决定（本次明确）：
>
> - **SFT 只覆盖 0 次 observe + 1 次 observe**（受限于现有数据生成 pipeline，每条最多 1 次 observe）；
> - **多次 observe 不再做单独的 SFT 数据**，全部交给 OPD 阶段；
> - 学生**最终输出形态恒定为**：
>
>   ```text
>   <think>
>   <latent block 1>           ← 奇数块（推理）
>   <latent block 2>           ← 偶数块（感知）  ← 仅当 block 1 末尾解码为 ".../>"
>   <latent block 3>           ← 奇数块（推理 / 收尾）
>   ...
>   </think>
>   <answer>...</answer>
>   ```
>
>   block 总数恒为奇数 N = 2k+1，k = observe 次数。

### 9.1 样本类型 A：0 次 observe（k = 0，仅 1 个奇数块）

**真实样本**（来自 NExT-QA / STAR）：

```json
{
  "question": "What did the person do with the food?\nOptions: (A) Put down. (B) Tidied up. (C) Threw. (D) Took.",
  "trajectory": "<think>\n[Analyze] What did the person do with the food?\n[Reason] Based on the video content, I can determine the answer.\n</think>\n<answer>(A) Put down.</answer>"
}
```

**块切分**：

| block | 奇/偶 | GT 文本（latent 解码目标） | 监督来源 |
|---|---|---|---|
| 1 | 奇 | `[Analyze] What did the person do with the food?\n[Reason] Based on the video content, I can determine the answer.` | **学生 SFT**：冻结 Decoder 解码后 CE 到这段 GT 文本 |

**学生学什么**：在 1 个奇数块内完成全部内省推理，最后冻结 Decoder 解码出 EOB（`\n` 或 EOS），且**末尾不出现 `/>`** → 状态机直接吐 `</think>`+`<answer>`。

**Teacher_R / Teacher_P 是否参与？**

- **Teacher_R**：本条样本**不进 Teacher_R 训练集**（教师 SFT 只用含 observe 的条目，专注教推理→工具调用→收束）。
- **Teacher_P**：本条样本不参与（无感知子任务）。

> 这类样本的速度收益：将原本 ~30 个推理 token（`[Analyze] ... [Reason] ...`）压缩为 < 30 个 latent step，且无需经过 LM head 反复采样 token，加速明显。

### 9.2 样本类型 B：1 次 observe（k = 1，3 个块：奇-偶-奇）

**真实样本**（来自 Charades-STA / temporal_locate）：

```json
{
  "video": ".../charades_sta/videos/TS75H.mp4",
  "question": "At what time in the video does \"a person holding medicine sneezes twice\" happen?",
  "trajectory": "<think>\n[Analyze] Need to localize when \"a person holding medicine sneezes twice\" happens.\n<observe type=\"temporal_locate\" target=\"a person holding medicine sneezes twice\"/>\n<result>It happens from 0.0s to 13.6s.</result>\n[Conclude] Time span confirmed.\n</think>\n<answer>0.0s-13.6s</answer>"
}
```

**块切分**：

| block | 奇/偶 | GT 文本（latent 解码目标） | 监督来源（SFT） | 监督来源（OPD） |
|---|---|---|---|---|
| 1 | 奇 | `[Analyze] Need to localize when "..." happens.\n<observe type="temporal_locate" target="..."/>` | 学生：冻结 Decoder CE 到此文本 | 学生：冻结 Decoder logits ∥ Teacher_R logits（KL）|
| 2 | 偶 | `It happens from 0.0s to 13.6s.` | 学生：冻结 Decoder CE 到此文本 | 学生：冻结 Decoder logits ∥ Teacher_P logits（KL）|
| 3 | 奇 | `[Conclude] Time span confirmed.` | 学生：冻结 Decoder CE 到此文本 | 学生：冻结 Decoder logits ∥ Teacher_R logits（KL）|

**关键约定**：

- block 1 的 GT 文本**必须以 `/>` 结尾**——这迫使学生在 SFT 阶段就学会"奇数块要内含完整 observe 调用"；
- block 2 的 GT 文本是 result 内的纯文本（**不含**外面的 `<result>` `</result>` 标签）；
- block 3 的 GT 文本不出现 `/>`——这教会状态机"该收尾了"。

**三条独立训练数据线**（同一条 GT trajectory 派生出三套独立的 (input, output) 对）：

```mermaid
flowchart TB
    GT["原始 GT trajectory"]
    GT --> S["学生数据：保留全部块\nblock1/3 用 latent_odd_mask + 冻结 Decoder CE\nblock2 用 latent_even_mask + 冻结 Decoder CE\n外壳 &lt;think&gt;/&lt;/think&gt;/&lt;answer&gt; 用学生 head CE"]
    GT --> R["Teacher_R 数据（标准文本 SFT）：\n输入=原问题；输出=&lt;think&gt;[Analyze]...&lt;observe.../&gt;&lt;result/&gt;[Conclude]...&lt;/think&gt;&lt;answer&gt;...\nresult 内容用 &lt;result/&gt; 占位，不学感知细节"]
    GT --> P["Teacher_P 数据（标准文本 SFT）：\n输入=视频/帧+子任务\"Locate when: a person holding medicine sneezes twice\"\n输出=&quot;It happens from 0.0s to 13.6s.&quot;"]
```

> **代码现状对照**：
> - 学生数据：`stage1_sft_train.py` 已有粗粒度 think_mask 双 head 路由；**待改造**为 §6.2-1 的 `latent_odd_mask` / `latent_even_mask` 双 mask；
> - Teacher_R 数据：`prepare_teacher_sft_data.py` 已实现 `<result/>` 占位替换；
> - Teacher_P 数据：`prepare_teacher_sft_data.py` 已实现 (子任务, GT 感知答案) 拆出。

### 9.3 SFT 阶段 forward 流程（学生视角，针对样本 B）

学生**输入序列**（也是被 mask 标注的序列）的 token 拓扑如下（`■` 表示该 token 实际进入学生输入；`□` 表示这是 GT 解码目标但学生输入仍是上一 token 的常规自回归方式）：

```text
位置        Token 内容                                              odd  even  使用的 head        监督到
────────────────────────────────────────────────────────────────────────────────────
[shell]    <think>\n                                              0    0     student head      下一 token

[block 1]  [Analyze] Need to localize when "..." happens.\n        1    0     frozen Decoder    GT: "<observe..."
           <observe type="temporal_locate" target="..."/>          1    0     frozen Decoder    GT: "/>" (块末)

[shell]    \n                                                     0    0     student head      "<result>"  ←可选

[block 2]  It happens from 0.0s to 13.6s.                         0    1     frozen Decoder    GT: result 内文本

[shell]    \n                                                     0    0     student head      "[Conclude]" ←可选

[block 3]  [Conclude] Time span confirmed.                        1    0     frozen Decoder    GT: 句末符

[shell]    \n</think>\n<answer>0.0s-13.6s</answer>                  0    0     student head      下一 token
```

> **L→T 与 L→L 边界关键**：
> - block 1 最后一个 latent step（位置在 `/>` 上）：冻结 Decoder 必须把"下一个 token"预测为**句末符 / EOS**，不再是别的内容——这正是 OPD 推理时状态机识别"该结束本块"的训练来源；
> - block 1 解码出来的整段文本以 `/>` 结尾——这是 OPD 推理时状态机识别"还要继续到偶数块"的训练来源；
> - block 3 最后一个 latent step：解码末尾**不**是 `/>`——状态机识别"推理结束，吐 `</think>`"。

### 9.4 OPD 阶段（合并版：单次 + 多次一起练）

#### 9.4.1 数据来源

- **不再单独制作多次 observe SFT 数据**；
- OPD 数据 = 复杂问题 + Teacher_R 现场 rollout 出的多次 observe 长链。
- 复杂问题来源举例（不需新数据集，仅需 LLM 重写问题）：
  - "Where is the red cup at the moment the person starts eating?"（先 temporal_locate 再 spatial_detect）
  - "What is the person doing after they pick up the red cup?"（先 temporal_locate 再 temporal_clip）

#### 9.4.2 训练流程（多次 observe 自然涌现）

```mermaid
sequenceDiagram
    participant Q as 复杂问题
    participant TR as Teacher_R (已 SFT)
    participant TP as Teacher_P (已 SFT)
    participant Tool as 专家工具
    participant S as 学生 (已 SFT，单次能力)

    Q->>TR: 接收问题
    TR->>TR: 文本空间 rollout：[Analyze]→<observe/>→<result/>→[Analyze]→<observe/>→<result/>→[Conclude]→<answer>
    Note over TR: 输出仅含 &lt;result/&gt; 占位，具体内容留待 Teacher_P 填
    loop 对 Teacher_R 输出的每个 <observe/>
        TR->>Tool: 触发工具
        Tool-->>TP: 工具产物 + 子任务
        TP->>TR: 生成感知文本，回填到对应位置
    end
    par 学生 OPD rollout
        S->>S: 状态机走块：奇 1 → (偶 2 → 奇 3) → (偶 4 → 奇 5) → ...
    and 教师双线监督
        TR-->>S: 奇数块 KL（每个奇数块对齐 Teacher_R 对应推理段）
        TP-->>S: 偶数块 KL（每个偶数块对齐 Teacher_P 对应感知段）
    end
```

#### 9.4.3 学生在 OPD 阶段补全的"新能力"

| 能力 | SFT 是否学过 | OPD 是否补全 | 监督来源 |
|---|---|---|---|
| 单次 observe 内的 [Analyze] 推理 | ✅ | — | — |
| observe 调用格式 `<observe type="..."/>` | ✅（在 block 1 末尾） | — | — |
| 感知段（block 2）内容生成 | ✅（监督到 GT result）| 升级为 KL 到 Teacher_P | Teacher_P |
| **第一个 observe 后是否启动第二个 observe** | ❌ | ✅ Teacher_R 在 block 3 写 `[Analyze]`+`<observe/>` 而不是 `[Conclude]`，KL 反传 | Teacher_R |
| **任意奇数 block 数（5、7、9…）** | ❌ | ✅ 状态机驱动 + Teacher_R KL | Teacher_R + 状态机硬上限 |
| 何时收束（最后一个奇数块不以 `/>` 结尾）| ✅ | ✅（同 SFT 机制 + Teacher_R 增强）| GT / Teacher_R |

#### 9.4.4 trajectory 形态对照（同一个复杂问题）

**Teacher_R 文本空间 rollout（监督目标，含 `<result/>` 占位）**：

```text
<think>
[Analyze] First locate when the person starts eating.
<observe type="temporal_locate" target="person starts eating"/>
<result/>                                                           ← 占位，由 Teacher_P 填
[Analyze] Now detect the red cup's position at 12.3s.               ← 第二轮推理！
<observe type="spatial_detect" frame="12.3" target="red cup"/>
<result/>                                                           ← 占位，由 Teacher_P 填
[Conclude] At 12.3s, the red cup is at [340,180,420,260].
</think>
<answer>At 12.3s when the person starts eating, the red cup is at [340,180,420,260].</answer>
```

**学生潜空间 rollout（同问题，5 个块）**：

```text
<think>
<block 1: 奇 - 推理>      ← KL ↔ Teacher_R 第一段 [Analyze]+<observe/>
                          ← 学生末尾解码为 ".../>" → 进入 block 2
<block 2: 偶 - 感知>      ← KL ↔ Teacher_P 对应 result 文本
                          ← 走完直接进 block 3
<block 3: 奇 - 推理>      ← KL ↔ Teacher_R 第二段 [Analyze]+<observe/>
                          ← 学生末尾解码仍为 ".../>" → 进入 block 4

                          ← 走完进 block 5
<block 5: 奇 - 收尾>      ← KL ↔ Teacher_R [Conclude] 段
                          ← 末尾解码不是 "/>" → 收尾
</think>
<answer>...</answer>      ← 学生 head CE / verifiable reward
```

### 9.5 数据规模与训练阶段对照

|---|---|---|---|
| Stage 1 SFT | 当前 36,852 条模板数据 | k=0（1 块）+ k=1（3 块） | 块数与每块长度由 GT 完全决定 |

### 9.6 取消多次 observe SFT 带来的简化

- ❌ 不再需要 `data_preparation/stage2_multistep_gen.py` 的 `--mode rule` / `--mode llm` 数据**专门做 SFT**；
- ✅ 保留这两个生成器作为 OPD 阶段**复杂问题来源**（trajectory 由 Teacher_R 现场 rollout，不预先标注）；
- ✅ 整条 pipeline 简化为 **Stage1 SFT（单次/0 次 observe）→ Stage2 OPD（单次蒸馏到潜空间 + 多次自发涌现）**。

## 十、最终一句话定调（覆盖 §八）

> **学生最终输出永远是 `<think>` + 奇数个 latent 块 + `</think>` + `<answer>`，控制符 `<observe/>` `<result>` 只存在于"冻结 Decoder 把 latent 解码出来给外部系统看"的过程中、不出现在学生 token 流里；SFT 用 GT 把每块解码内容锚定到语言空间，OPD 由状态机依据"奇数块末尾是否以 `/>` 结尾"动态决定块数，奇数块由 Teacher_R 监督、偶数块由 Teacher_P 监督。**
