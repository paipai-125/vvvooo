## 二、Observe 工具调用设计

系统在 `pipelines/__init__.py` 中定义了 **8 个工具 pipeline**，通过 `<observe type="..."/>` 指令由学生模型在 <think> 段中主动调用。

### 2.1 工具列表

| 工具名 | 预处理操作 | 输出给 Teacher_P 的内容 | 适用任务 |
|--------|-----------|------------------------|---------|
| `temporal_locate` | 在视频每帧左上角叠加时间戳 `t=XX.Xs`，底部加进度条 | 处理后视频 + 定位问题 | 时序定位（Charades-STA、DiDeMo） |
| `temporal_clip` | ffmpeg 裁切指定时间区间（前后各扩 1s 上下文），输出 8fps 小段 | 裁切后视频 + 描述问题 | 视频片段描述 |
| `spatial_detect` | 抽指定帧，用 Grounding-DINO 检测 `target` 物体并画绿色框 | 处理后图像 + 坐标输出问题 | 空间定位（VIPSeg、HC-STVG） |
| `spatial_crop` | 抽指定帧，按 bbox 裁切并扩大 20% 边距，长边放大到 768 | 裁切后图像 + 区域描述问题 | 区域内容描述 |
| `depth_overlay` | 抽指定帧，Depth-Anything V2 估计深度热力图 + Grounding-DINO 画物体框，叠加显示 | 处理后图像 + 空间关系问题 | 深度/空间关系问答 |
| `tracking_overlay` | 裁切时间区间，SAM 3 视频文本 prompt 跟踪，每帧叠加 mask 和 bbox 可视化 | 处理后视频 + 轨迹描述问题 | 运动轨迹描述（VIPSeg、HC-STVG） |
| `ocr_zoom` | 抽指定帧，按 bbox 裁切，Lanczos 放大到长边 1024 | 放大后图像 + OCR 问题 | 视频文字识别（TextVR） |
| `raw` | 无预处理，原视频原样返回 | 原视频 + 通用问题 | 无需特殊预处理的通用问答 |

### 2.2 工具参数规范

每个 `<observe>` 指令支持以下参数（不同工具必填参数不同）：

```
<observe type="工具名"
        target="描述文本（必填，除 raw 外）"
        time="时间区间，如 3.0-5.0（temporal_clip / tracking_overlay 必填）"
        frame="帧时间秒数（spatial_detect / spatial_crop / depth_overlay / ocr_zoom 必填）"
        bbox="[x1,y1,x2,y2] 像素坐标（spatial_crop / ocr_zoom 选填）"
        objects="逗号分隔的物体名（depth_overlay 必填）"
/>
```

### 2.3 工具调度入口

文件：`pipelines/__init__.py`

```python
PIPELINES = {
    "temporal_locate": temporal_locate_pipeline,
    "temporal_clip":   temporal_clip_pipeline,
    "spatial_detect":  spatial_detect_pipeline,
    "spatial_crop":    spatial_crop_pipeline,
    "depth_overlay":   depth_overlay_pipeline,
    "tracking_overlay": tracking_overlay_pipeline,
    "ocr_zoom":        ocr_zoom_pipeline,
    "raw":             raw_pipeline,
}

def run_pipeline(video_path: str, observe_query: ObserveQuery) -> dict:
    """根据 observe_query.type 分发到对应 pipeline"""
    pipeline_fn = PIPELINES.get(observe_query.type)
    if pipeline_fn is None:
        raise ValueError(f"未知工具类型: {observe_query.type}")
    return pipeline_fn(
        video_path=video_path,
        target=observe_query.target or "",
        time=observe_query.time,
        frame=observe_query.frame,
        bbox=observe_query.bbox,
        objects=observe_query.objects,
    )
```

## 三、如何利用数据集现有标注实现工具调用

### 3.1 核心思路

**不需要人工标注工具调用指令**。系统利用数据集自带的标注（时间区间、bbox、描述文本等）作为 ground-truth，通过规则模板自动生成含 `<observe>` 的完整 trajectory（思考链 + 工具调用 + 结果 + 答案）。

### 3.2 各数据集 → 工具调用的映射

#### Charades-STA / DiDeMo（时序定位）

**标注内容**：`(video_id, start, end, query)`

**生成逻辑**（`wrap_temporal_locate`）：
```
Ground-truth: start=5.0, end=10.0, query="person turn a light on"

生成的 trajectory:
<think>
[Analyze] Need to localize when "person turn a light on" happens.
<observe type="temporal_locate" target="person turn a light on"/>
<result>It happens from 5.0s to 10.0s.</result>
[Conclude] Time span confirmed.
</think>
<answer>5.0s-10.0s</answer>
```

**训练时**：`<result>...</result>` 中的内容为 ground-truth，用于计算 KL loss（学生输出的 logits 对齐到 Teacher_P 在该工具预处理后视频上的输出 logits）。

#### VIPSeg（空间定位 + 跟踪）

**标注内容**：全景分割标注，每帧含 124 类物体 bbox + instance_id（跨帧一致）

**空间定位**（`wrap_spatial_detect`）：
- 用类别名作为 `target`，用标注中的 bbox 作为 ground-truth 坐标
- 生成 `<observe type="spatial_detect" frame="XX.X" target="类别名"/>`

**区域描述**（`wrap_spatial_crop`）：
- 用标注中的 bbox 作为 `bbox` 参数
- 用类别名 + 简单描述作为 `gt_description`

**运动跟踪**（`wrap_tracking_describe`）：
- 利用跨帧 instance_id 一致性构建 bbox 序列
- 生成 `<observe type="tracking_overlay" time="start-end" target="描述"/>`
- `verifiable=False`（无精确坐标 ground-truth，由 SAM3+DINO 在 pipeline 中辅助）

#### HC-STVG v2（指代表达式定位）

**标注内容**：自然语言描述 + 时间区间 + 每帧 bbox

**用法**：作为 VIPSeg 的补充数据源，特别适合 `tracking_overlay`（人的跟踪 + 自然语言描述）

#### TextVR（OCR）

**标注内容**：帧时间 + 文字内容

**生成逻辑**（`wrap_ocr_zoom`）：
- 用标注中的帧时间作为 `frame`，文字内容作为 `gt_text`
- 如有 bbox 标注则一并写入 `<observe bbox="..."/>`

#### NExT-QA / STAR / CLEVRER（VQA）

**标注内容**：选择题（问题 + 选项 + ground-truth 答案）

**生成逻辑**（来源 B，LLM 改写）：
- 用 Qwen3-235B 将 ground-truth 答案改写为完整 trajectory
- 模型自主决定是否需要 `<observe>` 调用（LLM 生成的轨迹中可能含工具调用，也可能直接推理）

### 3.3 数据生成脚本

| 脚本 | 功能 |
|------|------|
| `data_preparation/stage1_sft_template.py` | 来源 A：规则模板生成 7 类工具样本（无需 LLM） |
| `data_preparation/stage1_sft_llm.ipynb` | 来源 B：LLM 改写 NExT-QA/STAR/CLEVRER 为轨迹数据 |
| `data_preparation/build_all_jsonl.py` | 将各数据集原始标注转为标准 `train.jsonl` 格式 |

## 四、训练/推理时根据 Observe 指令灵活调用工具

### 4.1 训练时：Trajectory 监督 + KL Loss

**文件**：`training/stage1_opd.py` — `train_one_sample_opd()`

**流程**：

```
输入: video + question（来自 JSONL 中的 trajectory 字段）

for each segment in trajectory:
    ┌─ OBSERVE 段 ──────────────────────────────────────┐
    │ 1. 解析 <observe type="..."/> 指令                  │
    │    → utils.parser.parse_observe()                   │
    │ 2. 调用 run_pipeline() 执行工具预处理               │
    │    → 得到处理后视频/图像 + perception_question       │
    │ 3. Teacher_P 在预处理后内容上输出 logits            │
    │    → teacher_p.get_segment_logits(...)              │
    │ 4. KL(学生 || Teacher_P) 计算损失                  │
    └─────────────────────────────────────────────────────┘
    ┌─ RESULT 段 ───────────────────────────────────────┐
    │ 将 <result> 内容作为上下文，继续计算 KL loss         │
    └─────────────────────────────────────────────────────┘
    ┌─ REASON 段 ───────────────────────────────────────┐
    │ Teacher_R 对推理链输出 logits，计算 KL loss         │
    └─────────────────────────────────────────────────────┘

最终 Loss = λ_kl·L_kl + λ_ans·L_ans + λ_aux·L_aux + λ_think_end·L_think_end
```

**关键点**：
- 训练时 `<observe>` 指令和 `<result>` 内容都来自 ground-truth 轨迹（JSONL 中固定），学生模型学习的是**在对应隐状态位置输出正确工具调用指令的分布**
- `verifiable=False` 的样本不计算 KL loss（无精确 ground-truth），只做格式监督

### 4.2 推理时：模型自主决策调用工具

**文件**：`training/stage1_opd.py`（推理模式）— 以及 `pipelines/run_pipeline()`

**流程**：

```
输入: video + question（用户提问）

 latent_engine.phase1_forward_efficient()  # 学生编码器
    ↓
 逐段生成 <think> 内容（Translator 自回归解码）
    ↓
 解析生成的文本，检测 <observe type="..."/> 
    ↓
 若存在 observe 指令:
    ├─ 调用 run_pipeline() 执行对应工具
    ├─ 将处理后视频 + perception_question 传给 Teacher_P
    └─ 将 Teacher_P 输出作为 <result> 内容，继续生成
    ↓
 若不存在 observe 指令（或已满足退出条件）:
    └─ 直接输出 <answer>...</answer>
```

**灵活调用的实现**：
1. **解析器**：`utils/parser.py` — `parse_observe()` 从生成文本中提取 `ObserveQuery` 数据结构（type, target, time, frame, bbox, objects）
2. **调度器**：`pipelines/__init__.py` — `run_pipeline()` 根据 `ObserveQuery.type` 分发到对应 pipeline 函数
3. **缓存机制**：每个 pipeline 用 `hash_key(video_path, 参数...)` 计算缓存键，避免重复处理

### 4.3 工具调用的参数来源

| 参数 | 训练时来源 | 推理时来源 |
|------|-----------|-----------|
| `type` | 模板函数固定写入 | 学生模型自主生成 |
| `target` | 数据集标注中的 query / 类别名 / 描述 | 学生模型根据问题分析自主生成 |
| `time` | 数据集标注中的 start/end | 学生模型根据前序观察结果推断 |
| `frame` | 数据集标注中的时间戳 / 随机采样 | 学生模型自主决定观察哪一帧 |
| `bbox` | 数据集标注中的 ground-truth bbox | 前序 `spatial_detect` 的输出结果 |
| `objects` | 数据集标注中的类别列表 | 学生模型根据问题分析自主生成 |

## 五、总结

```
数据集标注（GT）
    │
    ▼
stage1_sft_template.py ──→ 含 <observe> 的 trajectory（JSONL）
    │                        │
    │                        ▼
    │                  训练时: KL(T_φ∥S_θ) 监督
    │                  学生学习何时调用哪个工具
    │
    ▼
推理时:
  用户输入 video + question
    │
    ▼
  学生模型（Qwen3-VL-4B）自主生成 <think>...<observe type="..."/>...
    │
    ▼
  parse_observe() 解析指令
    │
    ▼
  run_pipeline() 执行工具预处理
    │
    ▼
  Teacher_P（Qwen3-VL-8B）在预处理后内容上感知
    │
    ▼
  <result> 内容返回给学生，继续推理
    │
    ▼
  输出 <answer>...</answer>
```

**设计优势**：
1. **零人工标注工具调用**：利用数据集自带标注自动生成轨迹
2. **灵活扩展**：新增工具只需实现对应 pipeline 函数并注册到 `PIPELINES` 字典
3. **缓存友好**：基于参数哈希的缓存避免重复预处理
4. **分级监督**：`verifiable=True/False` 区分精确监督和非精确监督样本
